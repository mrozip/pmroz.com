# pmroz.com

My personal website built with [Hugo](https://gohugo.io/) using YAML configuration following CI/CD DevOps best practices - Hosted on [Azure Static Web Apps](https://azure.microsoft.com/en-us/services/app-service/static/) and deployed via [GitHub Actions](https://github.com/features/actions).

Live at: [https://pmroz.com](https://pmroz.com)

---

## Project Structure

```
.github/
  workflows/
    azure-static-web-apps.yml   # CI/CD pipeline
    terraform.yml               # Azure infrastructure pipeline
archetypes/
  default.md                    # Hugo content archetype
assets/
  css/
    main.css                    # Site stylesheet
content/
  _index.md                     # Homepage content
  about.md                      # About me page
infra/
  terraform/                    # Azure resource group and Static Web App IaC
layouts/
  404.html                      # Custom 404 page
  index.html                    # Homepage template
  _default/
    baseof.html                 # Base HTML template
hugo.yaml                       # Hugo configuration
staticwebapp.config.json        # Azure Static Web Apps configuration
```

The `public/` directory is generated by Hugo during the build process and is not committed to the repository.

## Cost-Control Design

The site is intentionally designed to avoid Azure services that may introduce avoidable cost.

For this project's scope, do not add the following unless there is a clear reason:

* Azure App Service Plan
* Azure Container Registry
* Azure Container Apps
* Azure Virtual Machines
* Azure Front Door
* Azure CDN
* Application Insights
* Azure Blob Storage static website hosting
* paid database services
* server-side API services

The expected Azure resource group should contain only:

```
Resource group
└── Static Web App
```

A budget alert should be configured separately in Azure Cost Management.

## Azure Infrastructure

Terraform configuration lives in `infra/terraform` and provisions:

```
Resource group
└── Static Web App
```

Terraform is intended to run from GitHub Actions only:

- Manual workflow runs can plan, apply, destroy, and bootstrap the remote Terraform state backend.

The Terraform state storage account is a separate GitHub Actions backend dependency.

Terraform does not run automatically for pull requests or pushes to `main`. The normal PR and merge path is reserved for updating the existing Azure Static Web App through `.github/workflows/azure-static-web-apps.yml`.

To test a new resource group and Static Web App without touching the production state, run the `Terraform` workflow manually with:

| Input | Example |
| --- | --- |
| `operation` | `apply` |
| `bootstrap_state` | `true` on the first run only |
| `resource_group_name` | `rg-pmroz-com-sandbox` |
| `static_web_app_name` | `swa-pmroz-com-sandbox` |
| `state_key` | `sandbox.tfstate` |

Use the same `state_key` and names with `operation=destroy` to remove the sandbox resources.

Required GitHub variables:

| Variable | Purpose |
| --- | --- |
| `AZURE_CLIENT_ID` | Azure identity used by GitHub Actions OIDC |
| `AZURE_TENANT_ID` | Azure tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Azure subscription ID |
| `TF_STATE_STORAGE_ACCOUNT_NAME` | Azure Storage account for Terraform state |

Keep `AZURE_STATIC_WEB_APPS_API_TOKEN` pointed at the existing production Static Web App. Do not replace it with a sandbox deployment token unless you intentionally want the deployment workflow to target that sandbox app.

## Deployment Workflow

The site is deployed via `.github/workflows/azure-static-web-apps.yml`.

| Setting               | Reason                                             |
| --------------------- | -------------------------------------------------- |
| submodules: recursive | Required if the Hugo theme is a Git submodule      |
| extended: true        | Required for themes using Hugo Extended features   |
| hugo --minify         | Builds the production site with minified output    |
| app_location: public  | Deploys Hugo output only                           |
| skip_app_build: true  | Prevents Azure from trying to build the site again |

Pull requests targeting `main` create an Azure Static Web Apps preview environment for site changes. Pushes to `main` build and deploy those site changes to the existing Azure Static Web App.

## Azure Static Web Apps Configuration

`staticwebapp.config.json` controls Azure Static Web Apps behaviour:

- Routes 404 errors to the Hugo-generated `404.html` page
- Sets `X-Content-Type-Options` and `Referrer-Policy` security headers globally

---

## Required GitHub Secret

The deployment workflow requires one GitHub Actions secret:

**`AZURE_STATIC_WEB_APPS_API_TOKEN`**

To retrieve the token:

1. Azure Portal → Static Web App → **Manage deployment token**

To add it to the repository:

1. GitHub repository → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**
2. Name: `AZURE_STATIC_WEB_APPS_API_TOKEN`
3. Value: deployment token from Azure

## Local Development

Build the site locally with:

```bash
hugo server
```

Build a production output with:

```bash
hugo --minify
```

Output is written to `public/`.
